IntersectionObserver fruit slider

Here I'm demonstrating how IntersectionObserver allows you to be more declarative with your slide change logic and tracking. Instead of programmatically animating the slides we're using native DOM scrolling and simply observing where the slider is at.

See the Pen Fruit slider by Josh Howson (@josh-howson) on CodePen.

The result is a much simpler slide tracking strategy where we're not concerned with how to make the slide changes, rather we simply decide what to do when the slides do change.

Swipe-n-Tap

This one's also using IntersectionObserver. I came up with this method to natively drag & drop a switch, with fallbacks for mouse/keyboard folks too. It actually uses some trickery to give the impression that you're dragging and dropping, but in reality it's hacking native scroll behaviors to do reverse scrolling.

I have 3 elements: Spacer → Blue selection indicator → Spacer. As you scroll the element further to the right, the selection comes to the left, and vice-versa. When IntersectionObserver sees that you've scrolled more than halfway it updates the state and you can do whatever you want! In this case we're switching between two categories of results.

See the Pen Swipe n Tap by Josh Howson (@josh-howson) on CodePen.

This would be cumbersome to implement with JS. Drag event handling isn't easy, but our browsers take care of this behavior by default so why not leverage it?

CSS 62.5% font size converter

It's so much easier to work with rems when the root font size is 10px, or better yet: 62.5%.

Paste your whole stylesheet into this tool and it will multiply all rem values by 1.6 so you can finally switch from :root { font-size: 16px; } or :root { font-size: 100%; } to :root { font-size: 62.5% }.

See the Pen CSS rem converter to 62.5% by Josh Howson (@josh-howson) on CodePen.

Making borders overlap images

I find that designers often want their images to have outlines which overlay the photo. Not a border on the outside of the image, but right on top of the image with reduced opacity.

This is the default behavior in design tools like Figma, but it's not as simple on the web because borders live the outside of a box's content.

Recently browsers started allowing for rounded corners on the outline property, but in 2023 support isn't quite there yet. This is what I always do to solve this.

See the Pen Border overlapping image by Josh Howson (@josh-howson) on CodePen.

Sticky scroll to the bottom of the screen

I'm not sure how useful this one is, but it's an interesting exploration.

Have you ever tried making an element sticky to the bottom of the screen? It's not the typical behavior for position: sticky. Usually you want an element to be on the page, then when you scroll past it, rather than having it leave the page, it stays "stuck" to one of the edges of the scrolling element.

This is the opposite. Your element smoothly scrolls into view, and once it's 100% visible it sticks.

You usually need JS to make this work - usually you'll want to add a scroll event listener or use some other tool like IntersectionObserver, and after a certain scroll position you stick it to the bottom of your screen. This solution kind of accomplishes this using only CSS.

See the Pen insane by Josh Howson (@josh-howson) on CodePen.

As you may have noticed, the drawback is that once you scroll to the end the element gets lost and unsticks itself right in the middle of the page.

Not perfect but maybe this behavior could have a use case...
